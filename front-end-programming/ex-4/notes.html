Exercise 4: Vite Full Exercise

Objective

Use Vite to its full potential, implementing advanced features and best practices.

Steps

  1. Environment-Specific Configurations:
    • Set up .env.development and .env.production files to define separate API endpoints.
    • Task: Create an app that:
      • Fetches data from an endpoint specified in the environment variables.
      • Ensures the correct environment file is used during development (npm run dev) and production (npm run build).
  2. Advanced Vite Practices:
    • Use vite-plugin-checker for real-time TypeScript type-checking.
    • Use vite-plugin-eslint for linting during development.
    • Task: Add linting and type-checking tools to your Vite TypeScript project and display errors during the build process.
  3. Build Upon Previous Work:
    • Take the TypeScript application from Exercise 3 and:
      • Add advanced features like lazy loading modules and code splitting.
      • Optimize for production by ensuring minification and tree-shaking.
      • Test the performance of the final build.

My Approach

  1. Environment-Specific Configurations:
    • Created .env.development and .env.production files to store API endpoints: 
      .env.development
VITE_API_URL=https://dev-data.csv
                            
      .env.production
VITE_API_URL=https://prod-data.csv
    • Used import.meta.env.VITE_API_URL to fetch data dynamically based on the environment.
    • Tested the application by running npm run dev and npm run build.
  2. Advanced Vite Practices:
    • Installed plugins: 
      npm install vite-plugin-checker vite-plugin-eslint --save-dev
    • Configured Vite to use these plugins: 
      import { defineConfig } from 'vite';
import checker from 'vite-plugin-checker';
import eslintPlugin from 'vite-plugin-eslint';

export default defineConfig({
  plugins: [
    checker({ typescript: true }),
    eslintPlugin()
  ]
});
    • Tested real-time type-checking and linting during development.
  3. Build Upon Previous Work:
    • Implemented lazy loading and code splitting for modules: 
      import('./module.js').then(module => {
  module.doSomething();
});
    • Enabled tree-shaking and minification by ensuring the production build was optimized.
